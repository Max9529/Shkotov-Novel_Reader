# Novel Reader. Вимоги до вебзастосунку

## 1. Призначення системи

Novel Reader - це приватний вебзастосунок-читалка для зручного читання перекладених новел у форматі Markdown.

Застосунок має працювати як статичний сайт, розгорнутий через GitHub Pages. Контент новел зберігається у репозиторії як набір файлів: обкладинка, конфігураційний файл тайтлу та Markdown-файли окремих розділів. Додавання нових розділів або нових тайтлів виконується не через інтерфейс застосунку, а вручну через додавання файлів у репозиторій і push на GitHub.

Основний сценарій використання: користувач відкриває сайт із телефона, вибирає новелу, читає розділи, повертається до останнього місця читання, створює закладки та коментарі до абзаців, а також за потреби зберігає розділи для офлайн-читання.

## 2. Межі системи

### 2.1. Система повинна робити

Система повинна:

* показувати список доступних новел;
* відкривати сторінку конкретної новели;
* показувати обкладинку, назву, опис, прогрес читання та список розділів;
* завантажувати Markdown-файли розділів із репозиторію;
* відображати Markdown у зручному для читання вигляді;
* підтримувати навігацію між розділами;
* зберігати останній відкритий розділ;
* зберігати позицію читання всередині розділу;
* автоматично прокручувати розділ до останнього місця читання;
* дозволяти створювати закладки до конкретних абзаців;
* дозволяти створювати коментарі до конкретних абзаців;
* показувати окремий список коментарів у межах сторінки тайтлу;
* показувати статистику читання;
* дозволяти позначати розділи як "не почато", "читаю", "прочитано" та "улюблений";
* дозволяти змінювати тему оформлення;
* дозволяти налаштовувати параметри тексту під час читання;
* дозволяти експортувати та імпортувати локальні дані у JSON;
* дозволяти скинути локальні дані тільки після потрійного підтвердження та обов'язкового експорту JSON перед скиданням;
* підтримувати роботу на телефонах;
* підтримувати GitHub Pages;
* підтримувати офлайн-доступ до вибраних розділів після попереднього збереження.

### 2.2. Система не повинна робити

Система не повинна:

* мати бекенд;
* мати авторизацію;
* надсилати текст новели, прогрес, закладки або коментарі на сервер;
* використовувати зовнішню базу даних;
* редагувати Markdown-файли через інтерфейс;
* додавати нові розділи через інтерфейс;
* додавати нові новели через інтерфейс;
* виконувати автоматичну синхронізацію прогресу між пристроями;
* використовувати демо-дані;
* показувати рекламу;
* мати аналітику читання через сторонні сервіси.

## 3. Користувач системи

Єдиний цільовий користувач - власник приватного проєкту.

Користувач читає новели переважно з телефона. Також він може відкривати застосунок із ноутбука або іншого пристрою.

Користувач самостійно додає нові розділи та тайтли в репозиторій через push на GitHub.

## 4. Технічний контекст

Застосунок має бути реалізований як статичний клієнтський вебзастосунок для GitHub Pages.

Рекомендований стек:

* HTML;
* CSS;
* JavaScript;
* без серверної частини;
* без обов'язкових зовнішніх API;
* без фреймворків, якщо вони не потрібні для простої реалізації;
* localStorage для невеликих локальних налаштувань;
* IndexedDB або Cache API для локального кешу розділів для офлайн-читання;
* Markdown-парсер для відображення розділів.

Якщо використовується зовнішня бібліотека для Markdown, вона має бути підключена так, щоб застосунок стабільно працював на GitHub Pages. Бажано не робити критичну функціональність залежною від нестабільних CDN, якщо це можна уникнути.

## 5. Структура контенту

### 5.1. Загальна структура

У репозиторії має бути папка `ZFolder`.

У папці `ZFolder` мають зберігатися папки окремих новел. Назва папки новели має відповідати назві твору або стабільному slug твору.

Приклад структури:

```text
ZFolder/
  The Heavenly Demons Broadcast From the Demon Realm/
    cover.webp
    title.json
    chapter1_translation.md
    chapter2_translation.md
    chapter3_translation.md
    chapter4_translation.md
    chapter5_translation.md
    chapter6_translation.md
    chapter7_translation.md
    chapter8_translation.md
```

Допускається використання безпечних назв папок без апострофів і спеціальних символів, наприклад:

```text
ZFolder/
  heavenly-demon-broadcast/
    cover.webp
    title.json
    chapter1_translation.md
```

Рекомендований варіант - використовувати slug-папки латиницею, оскільки це зменшує ризик проблем із шляхами на GitHub Pages.

### 5.2. Конфігураційний файл тайтлу

Кожна новела повинна мати конфігураційний файл, наприклад `title.json`.

Приклад:

```json
{
  "id": "heavenly-demon-broadcast",
  "title": "The Heavenly Demon's Broadcast From the Demon Realm",
  "titleUk": "Трансляція Небесного Демона з Демонічного Царства",
  "author": "",
  "translator": "Claude",
  "cover": "cover.webp",
  "description": "",
  "language": "uk",
  "chapters": [
    {
      "id": "chapter-1",
      "number": 1,
      "title": "Небесний Демон Чі Юлан",
      "file": "chapter1_translation.md"
    },
    {
      "id": "chapter-2",
      "number": 2,
      "title": "Перша трансляція",
      "file": "chapter2_translation.md"
    }
  ]
}
```

### 5.3. Поведінка без повного конфігу

Якщо список розділів у конфігурації не заданий або заданий неповністю, застосунок повинен мати fallback-поведінку:

* спробувати знайти файли розділів за передбачуваним шаблоном назв;
* показати розділи як `Розділ 1`, `Розділ 2`, `Розділ 3` тощо;
* якщо в Markdown-файлі є заголовок першого рівня, використати його як назву розділу;
* якщо файл не знайдено, не ламати весь застосунок, а показати повідомлення про помилку для конкретного розділу.

Примітка: через обмеження статичного GitHub Pages браузер не може самостійно отримати список файлів у папці без заздалегідь заданого індексу. Тому для надійної роботи бажано підтримувати явний конфіг `title.json`.

## 6. Основні сутності

### 6.1. Novel

Поля:

* `id` - стабільний ідентифікатор новели;
* `title` - основна назва;
* `titleUk` - українська назва, якщо потрібна;
* `author` - автор, необов'язково;
* `translator` - перекладач або інструмент перекладу, необов'язково;
* `cover` - шлях до обкладинки;
* `description` - короткий опис, необов'язково;
* `language` - мова контенту;
* `chapters` - список розділів.

### 6.2. Chapter

Поля:

* `id` - стабільний ідентифікатор розділу;
* `number` - номер розділу;
* `title` - назва розділу;
* `file` - шлях до Markdown-файлу;
* `status` - локальний статус читання;
* `isFavorite` - локальна ознака улюбленого розділу.

### 6.3. ReadingProgress

Поля:

* `novelId`;
* `chapterId`;
* `paragraphId`;
* `scrollPercent`;
* `updatedAt`;
* `lastOpenedAt`.

### 6.4. Bookmark

Поля:

* `id`;
* `novelId`;
* `chapterId`;
* `paragraphId`;
* `chapterTitle`;
* `paragraphPreview`;
* `createdAt`.

Закладка прив'язується до абзацу, а не лише до розділу.

### 6.5. Comment

Поля:

* `id`;
* `novelId`;
* `chapterId`;
* `paragraphId`;
* `chapterTitle`;
* `paragraphPreview`;
* `text`;
* `createdAt`;
* `updatedAt`.

Коментар є окремою системою від закладок. Один і той самий абзац може мати закладку, коментар або обидва елементи одночасно.

### 6.6. ReaderSettings

Поля:

* `theme`;
* `fontSize`;
* `lineHeight`;
* `contentWidth`;
* `fontFamily`;
* `paragraphSpacing`;
* `textAlign`.

## 7. Головний екран застосунку

Головний екран застосунку показує всі доступні новели.

Оскільки зараз основна новела одна, інтерфейс має бути готовим до розширення на багато новел у майбутньому.

На головному екрані мають бути:

* назва застосунку `Novel Reader`;
* перемикач теми з іконкою сонця/місяця;
* список новел у вигляді карток;
* обкладинка кожної новели;
* назва новели;
* кількість розділів;
* загальний прогрес читання;
* кнопка `Відкрити`;
* кнопка `Продовжити`, якщо для новели вже є прогрес.

Якщо новела одна, її картка може займати центральне місце на сторінці.

## 8. Сторінка тайтлу

Сторінка тайтлу відкривається після вибору новели.

На сторінці тайтлу мають бути:

* обкладинка;
* назва новели;
* короткий опис, якщо заданий;
* кількість розділів;
* кількість прочитаних розділів;
* відсоток проходження;
* приблизний загальний час читання;
* кнопка `Продовжити читання`;
* список розділів;
* розділ `Закладки`;
* розділ `Коментарі`;
* розділ `Статистика`;
* налаштування для цього тайтлу;
* кнопка збереження розділів для офлайн-читання.

### 8.1. Рекомендований вигляд сторінки тайтлу

Рекомендований варіант:

* зверху компактний hero-блок із обкладинкою зліва або зверху на телефоні;
* поруч назва, прогрес і кнопка `Продовжити`;
* нижче вкладки:

  * `Розділи`;
  * `Закладки`;
  * `Коментарі`;
  * `Статистика`.

Цей варіант кращий за великий декоративний екран, бо основний сценарій - читання з телефона, де зайвий візуальний шум заважає.

## 9. Екран читання

Екран читання повинен бути максимально спокійним і мінімалістичним.

На екрані читання мають бути:

* назва розділу;
* текст розділу;
* кнопка повернення до тайтлу;
* кнопка відкриття налаштувань читання;
* кнопка перемикання теми;
* індикатор прогресу в розділі;
* кнопки `Попередній розділ` і `Наступний розділ`;
* можливість відкрити список розділів;
* можливість створити закладку до абзацу;
* можливість створити коментар до абзацу.

### 9.1. Поведінка під час відкриття розділу

Коли користувач відкриває розділ:

* якщо для цього розділу є збережений прогрес, застосунок автоматично прокручує текст до останнього абзацу;
* якщо прогресу немає, розділ відкривається з початку;
* якщо розділ відкрито через закладку або коментар, застосунок прокручує до відповідного абзацу;
* відповідний абзац має бути коротко підсвічений, щоб користувач побачив місце переходу.

### 9.2. Абзаци

Після рендерингу Markdown кожен абзац повинен отримувати стабільний локальний ідентифікатор.

Ідентифікатор може формуватися за номером розділу та порядковим номером абзацу, наприклад:

```text
chapter-3-p-42
```

Цей ідентифікатор використовується для:

* позиції читання;
* закладок;
* коментарів;
* переходу до місця в тексті.

Якщо текст розділу зміниться після оновлення файлу, старі закладки можуть зміститися. Це допустиме обмеження, але застосунок не повинен падати. Якщо абзац не знайдено, потрібно перейти до початку відповідного розділу і показати м'яке повідомлення.

## 10. Взаємодія з абзацом

Користувач повинен мати можливість взаємодіяти з конкретним абзацом.

На телефоні бажаний сценарій:

* користувач торкається абзацу або свайпає по ньому;
* з'являється компактне меню дій;
* у меню доступні:

  * `Закладка`;
  * `Коментар`;
  * `Скасувати`.

На ПК бажаний сценарій:

* користувач наводить курсор або клікає по абзацу;
* біля абзацу з'являється кнопка дій;
* після натискання відкривається меню `Закладка` / `Коментар`.

## 11. Закладки

Закладки повинні бути окремою системою.

Функціональні вимоги:

* користувач може створити закладку до конкретного абзацу;
* користувач може видалити закладку;
* один абзац не повинен отримувати дублікати однакової закладки;
* закладка повинна містити назву розділу і короткий фрагмент абзацу;
* список закладок має бути доступний на сторінці тайтлу;
* натискання на закладку відкриває відповідний розділ і прокручує до абзацу;
* абзац після переходу коротко підсвічується.

## 12. Коментарі

Коментарі повинні бути окремою системою від закладок.

Функціональні вимоги:

* користувач може створити коментар до конкретного абзацу;
* користувач може редагувати коментар;
* користувач може видалити коментар;
* коментар повинен містити текст користувача;
* коментар повинен зберігати короткий фрагмент абзацу;
* список коментарів має бути доступний на сторінці тайтлу;
* натискання на коментар відкриває відповідний розділ і прокручує до абзацу;
* абзац після переходу коротко підсвічується.

## 13. Статуси розділів

Для кожного розділу мають підтримуватися статуси:

* `Не почато`;
* `Читаю`;
* `Прочитано`;
* `Улюблений`.

Правила:

* якщо розділ жодного разу не відкривався, статус `Не почато`;
* якщо розділ відкрито і прогрес менший за поріг прочитання, статус `Читаю`;
* якщо користувач дійшов до кінця розділу або натиснув `Позначити як прочитано`, статус `Прочитано`;
* `Улюблений` є незалежною ознакою і може поєднуватися з іншими статусами.

Поріг автоматичного прочитання: 95% прокрутки розділу.

## 14. Прогрес читання

Система повинна зберігати:

* останню відкриту новелу;
* останній відкритий розділ;
* останній абзац у розділі;
* відсоток прокрутки розділу;
* дату останнього читання;
* загальний прогрес по новелі;
* кількість прочитаних розділів.

Прогрес повинен оновлюватися автоматично під час читання.

Оновлення прогресу не повинно відбуватися занадто часто. Рекомендовано використовувати debounce або throttle.

## 15. Статистика

На сторінці тайтлу потрібно показувати:

* загальну кількість розділів;
* кількість прочитаних розділів;
* кількість розділів у статусі `Читаю`;
* кількість улюблених розділів;
* кількість закладок;
* кількість коментарів;
* приблизну кількість слів;
* приблизний час читання;
* відсоток завершення новели;
* дату останнього читання.

Приблизний час читання можна рахувати за середньою швидкістю 180-220 слів на хвилину. Конкретне значення має бути стабільним у коді.

## 16. Пошук

Пошук по тексту не потрібен.

Не потрібно реалізовувати:

* пошук по поточному розділу;
* пошук по всій новелі;
* індексацію Markdown-файлів;
* підсвічування знайдених фрагментів.

## 17. Markdown-підтримка

Застосунок повинен підтримувати ті елементи Markdown, які використовуються в наданих файлах розділів.

Обов'язково підтримати:

* заголовки;
* абзаци;
* довгі текстові блоки;
* діалоги;
* горизонтальні розділювачі;
* списки, якщо вони є в тексті;
* жирний текст, якщо він є;
* курсив, якщо він є.

Markdown має рендеритися без редагування змісту.

Застосунок не повинен змінювати текст перекладу, автоматично виправляти слова або форматувати художній текст за власними правилами.

## 18. Налаштування читання

Користувач повинен мати можливість відкрити налаштування під час читання.

Налаштування:

* розмір шрифту;
* міжрядковий інтервал;
* ширина текстового блоку;
* тип шрифту;
* відступ між абзацами;
* вирівнювання тексту;
* тема оформлення.

Стандартні значення:

* розмір шрифту: 18 px на телефоні, 19-20 px на великих екранах;
* міжрядковий інтервал: 1.65;
* ширина текстового блоку: 680-760 px на великих екранах;
* вирівнювання: left;
* абзацний відступ: помірний;
* шрифт: системний або serif/sans-serif за вибором користувача.

Налаштування повинні зберігатися локально.

## 19. Теми оформлення

Інтерфейс повинен мати перемикач теми з іконкою сонця/місяця.

Мінімально потрібні теми:

* світла;
* темна.

Бажано також додати:

* сепія;
* темна фіолетова.

Перемикач теми має бути доступний:

* на головному екрані;
* на сторінці тайтлу;
* на екрані читання;
* у налаштуваннях читання.

Дизайн має бути мінімалістичним. Обкладинка може використовуватися як декоративний елемент, але не повинна створювати шум під час читання.

## 20. Обкладинка

Обкладинка новели використовується:

* на головному екрані у картці новели;
* на сторінці тайтлу;
* у метаданих тайтлу.

На екрані читання обкладинка не повинна займати багато місця. Допускається компактна мініатюра або повна відсутність обкладинки в режимі читання.

Не потрібно використовувати обкладинку як агресивний фон для тексту, бо це погіршує читабельність.

## 21. Офлайн-режим

Застосунок повинен підтримувати ручне збереження розділів для офлайн-читання.

Сценарій:

1. Користувач відкриває сторінку тайтлу.
2. Натискає `Зберегти для офлайн`.
3. Обирає:

   * всі розділи;
   * тільки непрочитані;
   * конкретні розділи.
4. Застосунок завантажує вибрані Markdown-файли та обкладинку.
5. Дані зберігаються локально в браузері.
6. Коли інтернету немає, застосунок відкриває збережені локальні копії.

Обмеження:

* офлайн-копії зберігаються тільки на конкретному пристрої;
* очищення даних браузера видаляє офлайн-копії;
* автоматичної синхронізації офлайн-копій між пристроями немає;
* якщо файл у репозиторії оновлено, користувач має вручну оновити офлайн-копію.

PWA-встановлення не є обов'язковим. Якщо для офлайн-режиму потрібен service worker, його можна реалізувати без вимоги робити повноцінний installable PWA.

## 22. Імпорт та експорт JSON

Система повинна підтримувати експорт локальних даних у JSON.

Експортувати потрібно:

* прогрес читання;
* статуси розділів;
* улюблені розділи;
* закладки;
* коментарі;
* налаштування читання;
* дату експорту;
* версію схеми даних.

Система повинна підтримувати імпорт JSON.

Під час імпорту:

* потрібно перевіряти валідність JSON;
* потрібно показати короткий підсумок того, що буде імпортовано;
* потрібно дати користувачу вибір:

  * замінити поточні дані;
  * об'єднати з поточними даними;
  * скасувати імпорт.

Якщо структура JSON не підтримується, потрібно показати зрозуміле повідомлення про помилку.

## 23. Скидання даних

Скидання локальних даних має бути заховане глибоко в налаштуваннях.

Перед скиданням система повинна:

1. Запропонувати експорт JSON.
2. Не дозволяти продовжити без експорту або явного підтвердження, що користувач відмовляється від експорту.
3. Показати перше підтвердження.
4. Показати друге підтвердження.
5. Показати третє підтвердження з текстом, що дія незворотна.

Після скидання видаляються:

* прогрес;
* закладки;
* коментарі;
* статуси;
* налаштування;
* офлайн-кеш, якщо користувач обрав повне скидання.

Потрібно підтримати два типи скидання:

* `Скинути прогрес і нотатки`;
* `Повне скидання з офлайн-кешем`.

## 24. Приватність

Застосунок приватний за призначенням.

Вимоги:

* немає бекенду;
* немає авторизації;
* немає серверної бази даних;
* немає аналітики;
* немає відправлення тексту новел на сервер;
* немає відправлення прогресу, закладок або коментарів на сервер;
* усі персональні дані читання зберігаються локально в браузері.

Важливе обмеження: GitHub Pages є статичним хостингом, а не базою даних. Він доставляє файли застосунку і Markdown-контент, але не зберігає користувацький прогрес.

## 25. Адаптивність

Основний формат використання - телефон.

Застосунок повинен коректно працювати на:

* телефонах;
* планшетах;
* ноутбуках;
* десктопах.

Вимоги для телефона:

* зручна ширина тексту;
* великі клікабельні кнопки;
* без горизонтального скролу;
* зручне нижнє або компактне меню;
* швидкий доступ до налаштувань;
* зручне створення закладок і коментарів;
* текст повинен залишатися читабельним без масштабування сторінки.

## 26. Навігація

На сторінці читання мають бути:

* кнопка повернення до сторінки тайтлу;
* кнопка `Попередній розділ`;
* кнопка `Наступний розділ`;
* список розділів;
* швидкий перехід до останньої позиції;
* відкриття закладок;
* відкриття коментарів.

Бокове меню можна реалізувати як висувну панель або нижній drawer на телефоні.

Рекомендована структура меню:

* `Розділи`;
* `Закладки`;
* `Коментарі`;
* `Налаштування`.

На телефоні краще використовувати нижню панель або drawer, а не класичний широкий sidebar.

## 27. Обробка помилок

Система повинна показувати зрозумілі повідомлення у таких випадках:

* не знайдено конфігураційний файл новели;
* не знайдено Markdown-файл розділу;
* не вдалося завантажити обкладинку;
* немає інтернету і розділ не збережено офлайн;
* JSON імпорту має неправильний формат;
* локальне сховище недоступне;
* офлайн-збереження не вдалося;
* браузер не підтримує потрібний механізм кешування.

Повідомлення мають бути українською мовою.

## 28. Валідація даних

Під час завантаження конфігурації потрібно перевіряти:

* наявність `id`;
* наявність `title`;
* наявність списку розділів або fallback-можливості;
* коректність шляхів до файлів;
* унікальність `chapter.id`;
* унікальність номерів розділів.

Під час створення коментаря:

* текст коментаря не може бути порожнім;
* надто довгий коментар має бути обмежений або попереджений;
* пробіли на початку та в кінці потрібно обрізати.

## 29. Збереження локальних даних

Ключі localStorage мають бути іменовані стабільно і з префіксом застосунку.

Приклад:

```text
novelReader.settings
novelReader.progress
novelReader.bookmarks
novelReader.comments
novelReader.chapterStatuses
novelReader.schemaVersion
```

Для більших даних, зокрема офлайн-копій Markdown-файлів, бажано використовувати IndexedDB або Cache API, а не localStorage.

Потрібно передбачити версію схеми даних для майбутніх змін.

## 30. Продуктивність

Застосунок повинен швидко відкриватися на телефоні.

Вимоги:

* не завантажувати всі розділи одразу без потреби;
* завантажувати текст розділу тільки під час відкриття або офлайн-збереження;
* не блокувати інтерфейс під час обробки довгого Markdown;
* не втрачати позицію читання під час переходів;
* не створювати надмірну кількість DOM-елементів без потреби;
* не використовувати важкі анімації на екрані читання.

## 31. Доступність

Вимоги:

* достатній контраст тексту і фону;
* видимі стани кнопок;
* підтримка клавіатурної навігації на ПК;
* зрозумілі підписи кнопок;
* кнопки не повинні складатися тільки з іконки без `aria-label`;
* текст має масштабуватися без руйнування верстки.

## 32. Критерії приймання

### 32.1. Головний екран

Застосунок вважається прийнятним, якщо:

* відкривається через GitHub Pages;
* показує назву `Novel Reader`;
* показує хоча б одну новелу з обкладинкою;
* дозволяє перейти до сторінки тайтлу;
* має перемикач теми.

### 32.2. Сторінка тайтлу

Прийнятно, якщо:

* показується обкладинка;
* показується назва новели;
* показується список розділів;
* показується прогрес;
* є кнопка `Продовжити читання`;
* є вкладки або секції `Розділи`, `Закладки`, `Коментарі`, `Статистика`;
* відкриття розділу працює.

### 32.3. Читання розділу

Прийнятно, якщо:

* Markdown-файл завантажується з репозиторію;
* текст коректно відображається українською;
* заголовки й абзаци зберігають структуру;
* є кнопки `Попередній розділ` і `Наступний розділ`;
* прогрес читання зберігається;
* після повторного відкриття розділ прокручується до останнього місця.

### 32.4. Закладки

Прийнятно, якщо:

* можна створити закладку до абзацу;
* закладка відображається на сторінці тайтлу;
* закладка веде до правильного розділу;
* після переходу сторінка прокручується до відповідного абзацу;
* закладку можна видалити.

### 32.5. Коментарі

Прийнятно, якщо:

* можна створити коментар до абзацу;
* коментар відображається у списку коментарів тайтлу;
* коментар можна редагувати;
* коментар можна видалити;
* натискання на коментар веде до відповідного абзацу.

### 32.6. Налаштування

Прийнятно, якщо:

* можна змінити тему;
* можна змінити розмір шрифту;
* можна змінити міжрядковий інтервал;
* можна змінити ширину текстового блоку;
* налаштування зберігаються після перезавантаження сторінки.

### 32.7. Офлайн

Прийнятно, якщо:

* користувач може вручну зберегти вибрані розділи для офлайн-читання;
* після втрати інтернету збережені розділи відкриваються;
* незбережений розділ без інтернету показує зрозуміле повідомлення;
* користувач може оновити офлайн-копії.

### 32.8. Експорт та імпорт

Прийнятно, якщо:

* можна експортувати локальні дані в JSON;
* JSON містить прогрес, закладки, коментарі, статуси та налаштування;
* можна імпортувати JSON назад;
* некоректний JSON не ламає застосунок;
* перед скиданням даних система пропонує експорт.

### 32.9. Скидання

Прийнятно, якщо:

* скидання знаходиться не на головному екрані, а в глибоких налаштуваннях;
* перед скиданням є потрійне підтвердження;
* перед скиданням є вимога експорту або явна відмова від експорту;
* після скидання дані справді видаляються.

## 33. Обмеження

* GitHub Pages не є базою даних.
* Автоматична синхронізація між пристроями без бекенду неможлива.
* Локальні дані зберігаються окремо на кожному пристрої.
* Очищення кешу або даних браузера може видалити прогрес, закладки, коментарі та офлайн-копії.
* Додавання нових розділів виконується через файли в репозиторії.
* Додавання нових новел виконується через нові папки в `ZFolder` і конфігураційні файли.
* Пошук по тексту не реалізується.
* Редагування Markdown через інтерфейс не реалізується.
* Демо-дані не потрібні.

## 34. Майбутнє розширення

Система має бути спроєктована так, щоб у майбутньому можна було додати десятки новел.

Для цього:

* логіка не повинна бути жорстко прив'язана до однієї новели;
* кожна новела повинна мати власний `id`;
* кожна новела повинна мати власний конфіг;
* локальні дані повинні прив'язуватися до `novelId`;
* закладки й коментарі повинні прив'язуватися до `novelId`, `chapterId` і `paragraphId`;
* список новел повинен формуватися з конфігураційного індексу.

Рекомендовано додати файл:

```text
ZFolder/index.json
```

Приклад:

```json
{
  "novels": [
    {
      "id": "heavenly-demon-broadcast",
      "folder": "heavenly-demon-broadcast",
      "config": "title.json"
    }
  ]
}
```

Це дозволить додавати нові новели без переписування основної логіки застосунку.

## 35. Рекомендований MVP

Перший робочий варіант повинен включати:

* GitHub Pages запуск;
* список новел;
* сторінку тайтлу;
* список розділів;
* відкриття Markdown-розділів;
* адаптацію під телефон;
* тему світла/темна;
* налаштування тексту;
* збереження прогресу;
* автопрокрутку до останнього абзацу;
* закладки;
* коментарі;
* JSON-експорт;
* JSON-імпорт;
* базове офлайн-збереження вибраних розділів.

Не потрібно в MVP:

* пошук;
* редагування тексту;
* авторизація;
* сервер;
* демо-дані;
* аналітика;
* складні анімації;
* повноцінна бібліотечна система з онлайн-керуванням файлами.
